1. Definisi Use Case Diagram

Konsep Utama

Diagram yang menggambarkan interaksi antara sistem dengan aktor luar. Fokus pada "Apa" yang dilakukan sistem, bukan "Bagaimana".

Menentukan batasan sistem (System Boundary).

Mengidentifikasi aktor yang berinteraksi dengan sistem.

Memetakan kebutuhan fungsional user secara visual.

Visualisasi Konsep Dasar

2. Komponen & Simbol

Actor

Entitas di luar sistem yang berinteraksi langsung. Bisa manusia, sistem eksternal, atau perangkat.

Stick figure / named role

Use Case

Deskripsi fungsi/fitur sistem yang memberikan nilai kepada aktor. Selalu berupa kata kerja aktif.

Ellipse / oval shape

System Boundary

Kotak bertanda nama sistem. Semua Use Case berada di dalam, semua Aktor di luar batas.

Dashed rectangle

Association

Garis solid yang menghubungkan Aktor dengan Use Case. Menunjukkan interaksi langsung.

Solid line with arrow

3. Relasi Include vs Extend

«include»

Wajib dieksekusi. Use Case A selalu memanggil Use Case B sebagai bagian dari prosesnya.

Contoh: "Beli Barang" selalu include "Login".

«extend»

Opsional / kondisional. Use Case B hanya diekseskusi jika kondisi tertentu pada A terpenuhi.

Contoh: "Bayar" di-extend "Cetak Struk" jika user meminta.

4. Generalization & Ringkasan Relasi

Generalization (Inheritance)

Use Case yang lebih spesifik mewarisi perilaku yang lebih umum. Panah terbuka (hollow arrowhead) dari child ke parent.

Ringkasan Semua Relasi

Association

Garis solid. Aktor ke Use Case. Interaksi langsung.

«include»

Putus-putus A→B. B wajib dijalankan oleh A.

«extend»

Putus-putus B→A. B opsional, hanya jika kondisi terpenuhi.

Generalization

Child→Parent. Pewarisan perilaku use case / aktor.

5. Step-by-Step Pengerjaan

Metodologi 5 Langkah

1
Identifikasi Aktor

Siapa yang berinteraksi dengan sistem? Manusia (user, admin) atau sistem eksternal (bank).

2
Identifikasi Use Case

Apa fungsi yang dibutuhkan setiap aktor? Kata kerja: Login, Tambah Produk.

3
Gambar System Boundary

Masukkan semua Use Case ke dalam kotak sistem. Aktor di luar batas.

4
Hubungkan Relasi

Tarik Association, lalu tambahkan include dan extend.

5
Verifikasi

(1) Aktor tidak ke aktor. (2) Semua UC di dalam boundary. (3) Setiap UC punya aktor.

6. Studi Kasus: Sistem ATM

Narasi Proses Bisnis

1. Nasabah memasukkan kartu dan PIN — sistem memproses Login.

2. Setelah login, Nasabah dapat melakukan Tarik Tunai atau Cek Saldo.

3. Setiap Tarik Tunai, sistem wajib menjalankan Update Saldo — include

4. Jika saldo tidak cukup, sistem menampilkan Tampilkan Error — extend

5. Sistem Bank Pusat sebagai aktor kedua memiliki akses ke Rekap Transaksi.

AKTOR

Nasabah · Bank Pusat

INCLUDE

Tarik → Update

EXTEND

Error → Tarik

8. Latihan: Reservasi Bioskop

Narasi Bisnis

Pengunjung memilih film dan memilih kursi. Saat membayar, pengunjung bisa memilih Gunakan Promo jika ada kode valid. Setelah bayar sukses, sistem otomatis mengirim tiket ke Email. Admin dapat mengelola jadwal film.

Aktor

Pengunjung · Admin

«include»

Bayar Tiket → Kirim Tiket Email

«extend»

Bayar Tiket ← Gunakan Promo

9. Post-Test Praktikum

Selesaikan diagram Use Case Reservasi Bioskop menggunakan draw.io dengan ketentuan:

STRUKTUR

  • Minimal 2 Aktor
  • System Boundary
  • Minimal 5 Use Case

RELASI WAJIB

  • Minimal 1 include
  • Minimal 1 extend
  • Asosiasi lengkap

VALIDASI

  • Aktor di luar kotak
  • Tidak ada garis menyilang
  • Label relasi jelas
